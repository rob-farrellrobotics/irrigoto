# Pinned Python build deps for irrigoto firmware compiles.
#
# Why pinned: ESPHome's beta channel (e.g. 2026.5.0b1) sometimes ships
# transitive-dep updates inside the same version label that introduce
# LWIP regressions (verified 2026-05-26 — a routine FW_BUILD bump
# started crashing in lwip_netconn_do_writemore until the host upgraded
# to 2026.5.1 stable). A plain `pip install esphome` re-resolves
# transitives at each install, which makes "works on Monday, broken on
# Wednesday" a real failure mode.
#
# Install (fresh venv recommended so global pip can't drift these):
#   python -m venv .venv
#   .venv\Scripts\activate          (Windows)  /  source .venv/bin/activate (POSIX)
#   pip install -r requirements.txt
#
# Bump intentionally: when you want to move to a newer ESPHome, edit
# this file, re-pin all three lines, compile, OTA, watch for ~5 min,
# then commit the bump as its own change.

# 2026-06-21: moved off the 2026.4.5 pin. HA Core updated to 2026.6.2, which
# bundles the 2026.6.x ESPHome *integration* (the native-API client). Firmware
# left at 2026.4.5 created a 2-minor API skew spanning the known-bad 2026.5.x
# reconnect-handshake changes -> ~30% cold-wake reconnect panics on both units
# (b437 baseline was 0/259 in mid-June, ~30% after the HA update). Matching the
# firmware to the integration (2026.6.2) closes the skew. Under test on f9e994;
# ba1f88 held on its running 2026.4.5 binary as the control.
esphome==2026.6.2
